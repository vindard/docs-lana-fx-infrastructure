Why not just add more currency pairs?

Act 2

Why FX Infrastructure Is Required

A multicurrency lending platform that holds BTC collateral has obligations a spot-rate converter cannot meet.

Four Reasons We Can't Bolt This On Later

Accounting Compliance

Fiat foreign-currency balances must be revalued at period-end closing rates (IAS 21 / ASC 830) — requiring historical rate records independent of any transaction. Every cross-currency entry must capture the exact rate used, because that rate becomes the baseline for all future revaluation deltas. Without both, financial statements cannot be produced correctly.

BTC and Fiat Are Different Regimes

BTC is an intangible asset at fair value (ASU 2023-08). Fiat FX uses closing-rate revaluation with unrealized/realized separation (IAS 21). These require separate account structures, separate revaluation logic, and must never be commingled. Treating BTC-to-USD the same as EUR-to-USD produces incoherent financials.

Custodied BTC = Agent Relationship

When the platform holds BTC collateral on behalf of borrowers, price movements affect both the asset and obligation equally — no P&L impact. This is legally distinct from BTC the platform owns. Commingling the two, or failing to reconcile on-chain balances against the ledger, creates the kind of segregation failure that brought down Celsius.

Operational Resilience

A single price feed with no staleness checks, no circuit breakers, and no fallback means one provider outage or one bad data point can silently corrupt every conversion, LTV calculation, and collateral valuation in the system. Rate infrastructure must be hardened before it underpins accounting entries.

Act 3

Two Accounting Regimes

BTC is not a foreign currency. This distinction drives every design decision.

Fiat FX vs BTC: Different Standards, Different Rules

Fiat FX  IAS 21 / ASC 830

  • Classification: Foreign currency
  • Record at spot rate on transaction date
  • Revalue monetary items at closing rate
  • Unrealized gains/losses → P&L or OCI
  • Period-end: delta or reversal method

BTC  FASB ASU 2023-08

  • Classification: Intangible asset at fair value
  • Fair value changes → net income (P&L)
  • No unrealized/realized distinction
  • Adjustments are cumulative, no reversal
  • Like trading securities under ASC 320
Key implication: BTC and fiat cannot share a trading account. Two regimes applied to one account balance produces incoherent financials. This is why the architecture separates them throughout.

BTC Collateral: The Agent Exception

When Lana holds BTC as agent (collateral on behalf of borrowers):

  • Asset (collateral held) and liability (obligation to return) move together
  • No P&L impact — the platform doesn't own this BTC
  • Revenue comes from interest income, not crypto appreciation
  • Revaluation still needed for LTV monitoring
BTC TypeRegimeP&L?
Custodied collateralAgentNo
Platform-ownedASU 2023-08Yes
Fee income (in BTC)ASU 2023-08Yes

This three-way split — fiat FX, platform BTC, custodied BTC — is why the architecture has three separate revaluation job chains.

C1 Rate Storage + C2 Multi-Source Aggregation

exchange_rates
├── from_currency   (BTC, USD, EUR)
├── to_currency
├── rate            DECIMAL(18,10)
├── rate_timestamp
├── rate_type       (SPOT, CLOSING)
└── source          (BITFINEX, COINBASE, ...)

Indexed by (pair, type, timestamp DESC) for fast lookups. Supports direct, inverse, and cross-rate triangulation via USD.

Aggregation Strategy

3+ independent sources → median (robust to outliers) → tolerance band filtering → store both aggregated + individual source rates

Safety Rails

  • BTC staleness: 60s (txn) / 5min (reporting)
  • Fiat staleness: 5min (txn) / 1hr (reporting)
  • Tolerance bands: BTC 2%, fiat 0.5% from median
  • Circuit breaker when sources < 2
  • Rate drift detection → client must re-quote

Rate Lookup: Direct → Inverse → Cross-Rate

async fn get_rate(from, to, at, rate_type) {
  // 1. Try direct pair
  if let Some(rate) = find_rate(from, to) {
    return Ok(rate);
  }
  // 2. Try inverse
  if let Some(rate) = find_rate(to, from) {
    return Ok(rate.invert());
  }
  // 3. Cross via USD triangulation
  let from_usd = find_rate(from, USD)?;
  let to_usd = find_rate(to, USD)?;
  Ok(from_usd.cross(to_usd))
}

Supports any pair via USD as pivot. Indexed by (pair, type, timestamp DESC).

C5 Period-End Revaluation

Three separate job chains, one event trigger. Each chain posts to its own accounts.

CoreTimeEvent::EndOfDay { day }
  │
  ├─── Fiat FX Revaluation (IAS 21)
  │      └── Collect fiat foreign-currency accounts → Process per account
  │          → Posts to 6100/6200 Unrealized FX Gain/Loss
  │          → Delta method (recommended): incremental, idempotent, no reversals
  │
  ├─── BTC Fair Value Reval (ASU 2023-08, platform-owned only)
  │      └── Collect platform-owned BTC accounts → Process per account
  │          → Posts to 7100/7200 BTC Fair Value Gain/Loss
  │          → Cumulative, no reversal
  │
  └─── Collateral Reval (agent, no P&L)
         └── Collect active facilities with collateral → Process per facility
             → Both-sides entry (asset + liability move together)
             → No P&L, for LTV monitoring + regulatory reporting

All three follow the established handler → collector → worker pattern (obix crate). Same pattern as interest accrual. Registered via outbox.register_event_handler() in FX module's init().

Segregation controls: Only authorized CALA templates can debit collateral accounts. Prevents commingling of customer and platform BTC.
Proof of reserves: Periodic reconciliation job compares on-chain custody wallet balances against ledger totals. Discrepancies alert immediately.
Cost basis: Each deposit is a separate lot (facility ID + timestamp). Per-facility tracking — no FIFO/LIFO assumptions. Needed for tax reporting on liquidation events.
Act 5

The Roadmap

3 independent groups, 5 phases + deferred. Build what delivers value first — let infrastructure emerge from real needs.

Implementation Phases

All groups are independent — no cross-group dependencies. Can run in any order or in parallel.

Group A — Audit Trail (independent, any workstream)

A
Rate-Per-Transaction Recording Audit Trail
Rate metadata on every CALA entry using existing PriceOfOneBTC feed. No new tables or services — can be bundled into Group B or done separately.

Group B — Fiat FX Accounting (phases run in parallel)

3
Fiat FX Trading Account + Realized Gain/Loss Per-Txn
Trading account (3200), realized gain/loss (4200/5100), conversion template. No jobs — fires inline on each fiat conversion.
4
Fiat FX Revaluation (Unrealized) Period-End
Unrealized gain/loss (6100/6200), delta method revaluation, EndOfDay job chain.

Group C — BTC Revaluation (parallel; Phase 5 depends on Phase 2)

2
Collateral Revaluation LTV Accuracy
Both-sides reval template (no P&L), EndOfDay job chain, existing price feed. Establishes collateral-vs-owned BTC boundary.
5
BTC Fair Value Revaluation BTC Ops
7100/7200 accounts, separate reval job chain. Depends on Phase 2 (collateral boundary). Needs ASU 2023-08 sign-off.
D
Rate Storage + Multi-Source Aggregation + Robustness
Full exchange_rates table, multi-provider (Bitfinex/Coinbase/Kraken), staleness enforcement, circuit breakers. Build when operational needs demand it.
D
Regulatory and Reporting
Audit trail enhancements, methodology docs, ASC 830 cash flow, proof of reserves. Consumers come after producers.
D
Segregation Controls + On-Chain Reconciliation
CALA template restrictions, on-chain/ledger recon job, proof-of-reserves attestation, cost basis / lot tracking. Enforcement layer on top of Phase 2 collateral accounting.
